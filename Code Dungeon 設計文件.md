# Code Dungeon（暫名）遊戲設計文件

版本：v0.1（2026-06-10）
參考作品：The Farmer Was Replaced、Bitburner、Screeps

---

## 1. 核心概念

一款「用程式操作勇者闖地牢」的 Roguelike 程式遊戲。玩家不直接操作角色，而是撰寫 JavaScript 腳本定義勇者的行為邏輯，勇者依腳本自動闖隨機生成的棋盤式地牢。

**核心賣點**：TFWR 教「自動化與最佳化」；本作教「在不確定環境下寫出穩健的決策邏輯」——條件處理、資源管理、優先序、防禦性思考。隨機地圖迫使程式碼泛化，這是與 TFWR（固定地圖）最大的差異。

**設計公式**：死亡重來 = debug cycle。程式品質 = 掛機效率。

## 2. 核心循環

1. 玩家撰寫/修改腳本（勇者行為邏輯）
2. 執行：勇者自動闖隨機生成的棋盤地牢
3. 死亡 → 觀察戰報與死因 → 修改程式 → 再闖
4. 通關樓層解鎖新 API（技能、語法、感知能力），永久保留

## 3. 遊戲結構（三層）

| 層級 | 時長 | 內容 |
|---|---|---|
| 單張地圖 | 3–10 分鐘 | 一個樓層，目標是找到樓梯口 |
| 一輪 Run | 約 30–90 分鐘，全自動 | 連續多層直到死亡或打倒最終 Boss；裝備、藥水、升級三選一全由程式決策 |
| Meta 循環 | 跨 Run | 依到達深度結算貨幣（暫稱「靈魂碎片」），解鎖新 API、新技能、基礎屬性 |

## 4. 程式語言：JavaScript

**選擇理由**：可直接做成網頁遊戲（零安裝、易傳播）；玩家學到真實世界最通用的語言。

**執行模型**：沙箱直譯器方案（如 JS-Interpreter）。玩家寫 `move("up")` 看起來同步，引擎在底層暫停執行。此方案天然獲得四項核心功能：

- 變速執行（1x–100x、無畫面瞬算）
- 逐步 debug（step-through）
- 無限迴圈防護（每 tick 指令數預算）
- 安全沙箱（玩家程式碼碰不到真實環境）

代價：支援語法為 ES5 子集，對遊戲腳本足夠。

**回合制**：每呼叫一次「行動函式」，遊戲推進一格時間（敵人也動一步）。確定性模擬，同 seed 必得同結果。

**自動重跑（採 TFWR 模式）**：腳本跑完自動從頭執行，直到死亡或通關。最簡腳本不需要主迴圈樣板；明確的 `while` 迴圈保留給多階段策略（先探圖→再撿補給→衝樓梯）。頂層變數每輪重置，跨輪狀態靠 `hp()` 等查詢函式。防呆：跑完一輪零行動的腳本直接停止並提示。

## 5. API 設計（三類）

### 感知（讀取狀態，不消耗回合）
`scan()`、`getEnemies()`、`hp()`、`mana()`、`getTile(x, y)`、`distance(target)`、`nearest(list)`、`hasPotion()`、`cooldown(skill)`

### 行動（消耗回合）
`move(dir)`、`moveToward(target)`、`attack(target)`、`fireball(target)`、`drinkPotion()`、`pickUp()`、`explore()`（自動探索未知區域）

### 運算（不消耗回合）
`if/else`、`for/while`、變數、函式定義、array、sort 等 JS 語法。

### 範例玩家程式碼

```javascript
// 腳本跑完自動從頭執行，不需要主迴圈
var enemies = getEnemies();
if (enemies.length > 0) {
  var target = nearest(enemies);
  if (mana() >= 30 && distance(target) <= 3) {
    fireball(target);
  } else {
    moveToward(target);
    attack(target);
  }
} else if (hp() < 50 && hasPotion()) {
  drinkPotion();
} else {
  explore();
}
```

## 6. 技能與裝備 = 語言的擴充

- **技能是函式**：`fireball(target)` 消耗魔力、有冷卻；玩家須在程式中管理資源。
- **武器改變函式行為**：弓 → `attack()` 變遠程；大劍 → 範圍 3 格但每兩回合才能行動。
- **裝備是被動修飾**：屬性加成、觸發效果。
- **語法本身是解鎖物**：初期只有 `move()`/`attack()` 與 `if`，逐步解鎖 `while`、函式定義、array——讓不會寫程式的人漸進學習。

## 7. Roguelike 元素

- **隨機地圖**：程式必須泛化，不能硬編路徑。
- **隨機掉落/事件**：由程式判斷取捨（如 `onItemFound(item)` 回呼）。
- **升級三選一**：以 `chooseUpgrade(options)` 回呼讓 build 決策也可程式化，自動化 meta 決策是高階玩法。
- **永久進度**：API 與語法解鎖跨輪保留；單輪內裝備、藥水死亡即失。

## 8. 放置（Idle）模式

核心開關：**死亡後自動重開新 Run**。打開後「程式品質 = 掛機效率」——寫得越穩健，平均闖得越深，每小時收益越高。

配套設計：

- **變速與背景模擬**：確定性回合制可 1x 觀看、100x 快轉、或無畫面瞬算；掛機跑無畫面模式，回來看戰報。
- **離線進度**：以同一套模擬器補算離線期間的 Run 數與收益（確定性回合制使此功能幾乎免費）。
- **戰報/死亡回放**：死因排行、平均到達樓層、關鍵回合回放——玩家據此決定改哪段程式。
- **防止無腦掛機**：深層樓層引入新機制，舊程式會卡住，迫使玩家回來迭代（留存循環）。

## 9. 難度曲線 = 程式概念教學曲線

| 樓層機制 | 逼出的程式概念 |
|---|---|
| 單一敵人直線走廊 | 順序執行、`if` |
| 多敵人 | 迴圈、目標排序（先打血少的？） |
| 迷宮 | 探索演算法（或解鎖內建 `moveTo(x,y)` 當捷徑） |
| 毒沼/陷阱地形 | 條件判斷地格、代價權衡 |
| Boss 行為模式 | 狀態機思維 |

## 10. 風險與對策

| 風險 | 對策 |
|---|---|
| 尋路對新手太難 | `moveTo(x, y)` 做成中期可解鎖的內建函式；高手可自寫 BFS 提早通過 |
| 隨機性導致挫折（沒寫錯但運氣差） | 保留 seed 重跑；平衡上讓夠好的程式有 90%+ 通關率 |
| 最佳解收斂（人人同一份程式） | 隨機 build 三選一 + 每日挑戰（固定 seed 排行榜，比回合數/程式行數） |
| meta 數值太強，退化成普通放置遊戲 | meta 成長以「解鎖能力」為主、數值加成為輔，寫更好的程式永遠是主要進步來源 |
