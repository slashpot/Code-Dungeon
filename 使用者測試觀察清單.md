# Code Dungeon 使用者測試觀察清單

目的：驗證核心玩法（寫腳本操控勇者）**好不好玩、新手學不學得會**。
用法：每位受測者印一份。你只觀察＋記錄，**全程不出聲、不提示**（被求助時記下「在這裡卡住」再給最小提示）。

---

## 0. 測試前準備

- [ ] 受測者組合：至少 1 位會寫程式、2 位不會
- [ ] 每人 15–20 分鐘，環境安靜、螢幕你看得到
- [ ] 開新分頁 / 清空存檔（存檔綁瀏覽器，避免被前一位的進度污染）
- [ ] 開錄影或錄螢幕（事後回看卡點）；先取得同意
- [ ] 開場只說一句：「這是一款用 JavaScript 寫腳本操控勇者闖地牢的遊戲，你試著玩玩看，我不會出聲。」

## 3. 逐關觀察（L1–L6）

每關記：**過關了嗎 / 卡點是「不會寫程式」還是「不懂遊戲規則」 / 情緒反應**。

各關重點觀察：

- **L1**：能否自己發現「腳本跑完會自動重來」，而不是手動寫迴圈？
- **L2**：有沒有意識到要先 `drinkPotion` / 規劃路線繞毒沼？還是直接送死才懂？
- **L3**：會用 `explore` / 偵察邏輯，還是亂走？「視野有限」這個設計有沒有讓他想寫偵察？
- **L4**：有沒有想到「先打誰」的優先序？怎麼表達的？
- **L5**：選擇硬拚囤藥水，還是繞 Boss 衝樓梯？有沒有「原來可以這樣」的驚喜？
- **L6**：卡住時走哪條路？（→ 詳記於第 4 節「修煉場」）

## 4.5 修煉場與 skill 系統（回答「軟門檻是否成立」）

這節是本輪測試的**核心決策資料**：skill 目前是軟門檻（不修煉也能靠手寫繞過），要不要收緊取決於玩家實際行為。

- [ ] 有沒有**自己注意到** skillbar？（什麼時候、什麼契機）點過 chip 嗎？
- [ ] 看得懂「skill 等級」和「修煉可升級」的概念嗎？（✓/◐/●/🔒 圖示有沒有讀懂）
- [ ] 在 L6 卡住時，走哪條路？
  - [ ] A. 進修煉場練 distance/nearest（設計期望路徑）
  - [ ] B. 在主腳本手寫等效邏輯（合法繞過——記下他寫了什麼）
  - [ ] C. 兩者都沒想到，反覆送死 / 放棄（hint 引導失敗）
- [ ] 若走 A：修煉題目描述看得懂嗎？testcase 失敗訊息有幫助嗎？通過後有沒有「我的程式碼變成勇者能力」的理解與成就感？
- [ ] 若走 B：他知道修煉場的存在嗎？（知道但嫌麻煩 vs 根本沒發現——前者是定位問題、後者是引導問題）
- [ ] 路徑用時：A ____ 分 / B ____ 分；情緒（有趣/像作業/煩）：____

> 判讀：B 不算失敗——手寫等效邏輯代表學習目標已達成。真正的警訊是 C（引導失敗）或 A 被評為「像作業」（修煉場定位失敗）。

## 整理：跨受測者彙整

測完 3 人後，回答這三個決策問題：

1. **核心玩法成立嗎？** 不會寫程式的人有沒有至少過 L1–L2 並露出成就感？
2. **最大瓶頸是教學、API、還是難度曲線？**（看卡點集中在哪）
3. **下一步該投資哪裡？** 若卡在入門 → 先做新手引導；若卡在中段無聊 → 先做技能/隨機地圖增加變化；若都順 → 放心推進大功能。
4. **skill 軟門檻成立嗎？**（看第 4.5 節）多數人走 A 或 B → 維持軟門檻；多數人落在 C → 問題在引導不在門檻，先改 hint/UI 而非加硬鎖。
